# Feature Specification: CPU 工作区 MVP

**Feature Branch**: `001-cpu-workspace-mvp`

**Created**: 2026-07-22

**Status**: Draft

**Input**: 从《异构算力容器教学服务平台设计方案》收敛第一个可交付功能：
为已授权用户提供可持久化的 CPU 工作区全生命周期管理，并提供基本项目范围权限、
操作进度与审计能力。

## Scope

本功能交付单节点环境中的 CPU 工作区闭环。用户可在所属项目内申请并管理自己的
工作区；项目负责人可管理项目成员并处理项目内工作区；平台管理员可管理全局配置。

以下能力不属于本功能：GPU、NPU 或其他加速器调度，多节点或多集群调度，课程/班级教学流程，
教师协作查看终端，Web 终端，镜像构建，数据快照，对象存储，学生服务发布，公网访问，多集群
资源代理，以及完整生产级监控和灾难恢复。后续功能不得在没有迁移方案的情况下改变本功能已经
发布的工作区状态含义、操作追踪行为、权限语义、数据保留选择或审计记录语义。

## Clarifications

### Session 2026-07-22

- Q: 默认重置完成后，工作区应处于什么状态？ → A: 重新准备运行环境，清除非持久化运行内容并保留持久数据；成功后工作区进入“运行中”。
- Q: 创建操作进行期间，用户失去项目成员资格、创建权限或可用额度时，应如何处理？ → A: 无论操作进度如何都立即终止，并尝试删除本次操作产生的所有资源和数据。
- Q: 删除工作区并选择“保留数据”后，保留的数据应如何处理？ → A: 数据继续归属原项目和原所有者并计入项目额度；所有者可将其绑定到同项目内自己的新工作区或单独删除，所有者离开项目后由项目授权人员处理。
- Q: 创建请求被接受后，所选工作区配置被停用或撤销时，应如何处理？ → A: 普通停用只阻止新请求，已接受的创建继续；安全或合规撤销立即终止创建并清理本次操作产生的资源和数据。
- Q: 用户选择“同时删除数据”，但运行环境已删除而持久数据删除失败时，应如何表示和恢复？ → A: 删除操作持续保持执行中并自动重试，直到数据删除成功，不允许用户发起显式重试。
- Q: 同一所有者在同一项目中提交重复工作区名称时，应如何处理？ → A: 未删除工作区及进行中的创建请求名称必须唯一；重复提交时拒绝并提示更换名称，已删除工作区不占用名称。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建并管理自己的 CPU 工作区 (Priority: P1)

作为项目成员，我希望创建一个经过批准的 CPU 工作区，并能查看其进度、停止、重新启动、
重置和删除它，以便在不重新配置环境的情况下持续完成开发或实验任务。

**Why this priority**: 这是平台的最小用户价值闭环；没有可控且可恢复的工作区，后续教学、
算力和终端能力没有可靠的承载对象。

**Independent Test**: 为一个已授权项目成员创建工作区，完成启动、停止、重启、重置和删除，
并验证每一步有可见结果、失败可理解、数据保留行为符合用户选择。

**Acceptance Scenarios**:

1. **Given** 项目成员有可用的工作区额度，**When** 选择一个批准的 CPU 工作区配置并提交创建，
   **Then** 系统提供唯一的操作追踪信息，并在成功后显示可用工作区。
2. **Given** 工作区正在运行，**When** 用户停止它，**Then** 系统显示已停止且用户数据仍可在随后
   启动时使用。
3. **Given** 工作区已停止或运行中，**When** 用户执行默认重置，**Then** 系统重新准备运行环境、清除
   非持久化运行内容、保留独立的用户持久数据，并在成功后将工作区置为运行中。
4. **Given** 用户提交相同的创建或生命周期请求两次，**When** 系统处理请求，**Then** 只产生一个
   对应的工作区或生命周期操作。
5. **Given** 创建操作尚未完成，**When** 用户失去项目成员资格、创建权限或可用额度，**Then** 系统
   立即终止该操作，尝试删除本次操作产生的全部资源和数据，并向授权人员显示最终处理结果。
6. **Given** 用户删除工作区并选择保留数据，**When** 工作区删除完成，**Then** 数据保持原项目和
   原所有者归属并继续计入项目额度，所有者可将其绑定到同项目内自己的新工作区或单独删除。
7. **Given** 创建请求已经被接受，**When** 所选配置被普通停用，**Then** 该创建可继续且系统拒绝
   后续使用该配置的新请求。
8. **Given** 创建请求尚未完成，**When** 所选配置因安全或合规原因被撤销，**Then** 系统立即终止
   创建并清理本次操作产生的资源和数据。
9. **Given** 用户选择同时删除工作区和持久数据，且运行环境已删除但数据删除失败，**When** 系统
   继续处理删除，**Then** 该删除操作保持执行中并自动重试至数据删除成功，用户不得另行发起该
   操作的显式重试。
10. **Given** 同一所有者在同一项目中已有同名的未删除工作区或同名创建请求正在进行，**When** 用户
    提交该名称的创建请求，**Then** 系统拒绝请求并提示用户更换名称；已删除工作区不占用名称。

---

### User Story 2 - 按项目范围管理访问 (Priority: P2)

作为项目负责人，我希望管理项目成员并查看或处理项目内工作区，以便在不暴露其他项目数据的
情况下支持团队成员。

**Why this priority**: 基础访问边界是共享教学和项目环境安全运行的前提；仅有登录无法防止
跨项目或跨用户访问。

**Independent Test**: 创建两个项目和不同成员，在其中一个项目执行成员与工作区管理操作，验证
另一个项目成员无法查看、修改或推测这些资源。

**Acceptance Scenarios**:

1. **Given** 项目负责人拥有成员管理权限，**When** 添加、移除或调整项目成员职责，**Then** 变更仅
   影响该项目并被记录。
2. **Given** 项目负责人或经授权的操作人员，**When** 查看或处理项目内成员的工作区，**Then** 系统
   显示操作主体、目标与结果。
3. **Given** 普通成员或另一项目成员，**When** 尝试访问不属于自己的工作区，**Then** 系统拒绝请求且
   不泄露资源是否存在。

---

### User Story 3 - 跟踪失败并安全恢复 (Priority: P3)

作为用户或项目负责人，我希望了解工作区操作是否失败、为何失败以及何时可以重试，以便从暂时性
故障中恢复而不丢失数据或产生重复资源。

**Why this priority**: 工作区依赖外部运行环境；可见的失败、重试和恢复能力将人工排障从常规流程中
移除。

**Independent Test**: 在创建、启动、停止或删除过程中模拟暂时性失败，验证系统保留操作历史、
显示可操作的状态，并在安全重试后不产生重复资源或错误覆盖新状态。

**Acceptance Scenarios**:

1. **Given** 一个工作区操作无法完成，**When** 系统确认失败，**Then** 用户可看到失败状态、可理解的
   原因和下一步操作。
2. **Given** 一个可恢复的失败操作，**When** 授权用户重试，**Then** 系统继续或重新执行该操作而不创建
   重复资源。
3. **Given** 用户已发起新操作，**When** 较早操作晚到的结果返回，**Then** 系统保留较新的工作区状态。
4. **Given** 调用者重复提交同一业务请求，**When** 系统收到重复提交，**Then** 系统返回原操作的追踪
   信息和结果，不产生新的副作用。
5. **Given** 一个允许恢复的失败操作，**When** 授权用户发起显式重试，**Then** 系统创建关联原失败
   操作的新操作记录，而非重复使用原请求。

### Edge Cases

- 用户在已有未完成操作时再次请求冲突的生命周期操作。
- 同一所有者在同一项目中提交与未删除工作区或进行中的创建请求同名的工作区；系统必须拒绝请求
  并提示更换名称，已删除工作区不占用名称。
- 用户在创建操作期间失去项目成员资格、创建权限或可用额度；操作必须立即终止，并清理本次操作
  已产生的全部资源和数据。
- 用户选择删除工作区但要求保留其数据；数据继续归属原项目和原所有者并计入项目额度，可由所有者
  重新绑定到同项目内自己的新工作区或单独删除，所有者离开项目后由项目授权人员处理。
- 用户选择删除工作区并同时删除数据；系统必须分别显示运行环境与持久数据的最终处理结果。
- 运行环境已删除但持久数据删除失败；删除操作保持执行中并自动重试至数据删除成功，不允许用户
  对该删除操作发起显式重试。
- 系统在外部环境已完成部分处理后中断或重启。
- 外部请求超时但资源可能已创建成功，或后台消息被重复处理；系统必须协调现有结果而不重复创建
  资源或产生重复业务副作用。
- 较旧工作区代次的结果在新操作之后返回；较旧结果不得覆盖较新状态。
- 项目负责人尝试访问另一项目的工作区，或普通成员猜测其他用户的资源标识；请求必须被拒绝且
  不泄露资源存在性。
- 用户对不可恢复失败重复重试；系统必须拒绝该重试并保留原失败原因。
- 用户尝试用无效、过期或不再被批准的工作区配置创建环境；普通停用只阻止新请求，安全或合规
  撤销还会立即终止尚未完成的已接受创建并清理本次操作产生的资源和数据。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将每个外部身份稳定映射到唯一的平台用户；电子邮件地址 MUST NOT 作为
  唯一或权威的用户身份依据。
- **FR-002**: 系统 MUST 支持平台管理员、项目负责人、项目操作人员、项目普通成员和项目查看者的
  职责；工作区所有权 MUST 是独立于项目职责的资源关系。所有职责 MUST 仅在其被授予的项目范围内
  生效。
- **FR-003**: 系统 MUST 仅允许拥有创建权限且额度可用的项目成员，从平台批准且当前项目已启用的
  CPU 工作区配置中选择并创建自己的工作区；用户 MUST NOT 任意指定运行环境、资源量或特权能力。
- **FR-004**: 系统 MUST 为创建、启动、停止、重置和删除请求生成稳定、唯一且可查询的操作标识，
  并显示等待、执行、成功或失败等当前状态、终态结果及安全的失败原因。
- **FR-005**: 系统 MUST 仅允许合法的工作区状态转换；冲突的生命周期操作 MUST NOT 同时改变或破坏
  工作区状态，且系统 MUST 告知用户现有操作尚未完成。
- **FR-006**: 系统 MUST 在停止后保留用户持久数据；默认重置 MUST 重新准备运行环境、清除非持久化
  运行内容、保留独立的用户持久数据，并在成功后使工作区进入运行中；删除请求 MUST 明确展示数据
  保留或删除的选择与结果。
- **FR-007**: 系统 MUST 要求用户同时具备相应操作权限和目标资源的所有权或项目范围授权；界面隐藏
  不得被视为访问控制。
- **FR-008**: 系统 MUST 将同一调用者对同一业务请求的重复提交视为同一操作，返回同一操作结果，
  且不得产生重复资源或重复副作用；用户对允许恢复的失败操作发起显式重试时，系统 MAY 创建新的
  操作记录，但 MUST 与原失败操作关联。
- **FR-009**: 系统 MUST 记录工作区创建、删除、重置、跨用户操作、项目成员变更和拒绝访问的主体、
  目标、时间、结果和关联追踪信息。
- **FR-010**: 系统 MUST 在工作区操作失败时保留失败历史、提供不暴露凭证、堆栈、数据库信息或
  运行环境敏感细节的原因，并仅在安全时允许授权用户重试或恢复；不可恢复失败的重复重试 MUST
  被拒绝并保留原失败原因。
- **FR-011**: 系统 MUST 只允许使用平台批准的工作区配置；用户不得指定任意运行环境或获得超出其
  配额的资源。
- **FR-012**: 系统 MUST 允许项目负责人查看项目内工作区的状态和操作历史，并禁止其访问其他项目
  的同类信息。
- **FR-013**: 创建操作进行期间，如果调用者失去项目成员资格、创建权限或可用额度，系统 MUST
  立即终止该操作，并尝试删除本次操作产生的全部资源和数据；清理进度、失败和最终结果 MUST
  保持可查询并形成审计证据。
- **FR-014**: 删除工作区并选择保留数据后，系统 MUST 保持数据的原项目和原所有者归属并继续计入
  项目额度；所有者 MUST 能够将其绑定到同项目内自己的新工作区或单独删除；所有者离开项目后，
  该数据 MUST 由项目授权人员处理。
- **FR-015**: 工作区配置被普通停用时，系统 MUST 拒绝使用该配置的新创建请求，并 MUST 允许已经
  接受的创建继续；配置因安全或合规原因被撤销时，系统 MUST 立即终止尚未完成的相关创建，清理
  本次操作产生的资源和数据，并使终止、清理及最终结果保持可查询和可审计。
- **FR-016**: 当用户选择同时删除数据且运行环境已删除、持久数据删除失败时，系统 MUST 保持删除
  操作处于执行中并自动重试至数据删除成功；用户 MUST NOT 对该删除操作发起显式重试，且自动处理
  过程必须可查询、可关联并可审计。在数据删除成功前，持久数据 MUST 保持原所有权、项目归属和
  额度影响。
- **FR-017**: 系统 MUST 为已接受但尚未执行、已执行但尚未产生外部副作用、以及已产生部分外部
  副作用的操作分别提供可追踪的终止或协调规则；平台进程中断、外部请求超时或消息重复处理后，
  系统 MUST 协调实际结果至明确状态，不得盲目重复创建资源。
- **FR-018**: 较早操作或较旧工作区代次的晚到结果 MUST NOT 覆盖较新的工作区状态。
- **FR-019**: 同一所有者在同一项目内，未删除工作区及进行中的创建请求的名称 MUST 唯一；系统
  MUST 拒绝重复名称的创建请求并提示用户更换名称。已删除工作区 MUST NOT 占用名称。

### Key Entities *(include if feature involves data)*

- **User**: 具有稳定平台身份、状态与平台级职责的个人或受控自动化主体。
- **Project**: 成员、额度、工作区配置和资源归属的最小管理范围。
- **Project Membership**: 用户在项目内的职责和有效访问范围。
- **Workspace Profile**: 平台批准的 CPU 工作区选项，包含用户可见名称、资源额度和可用状态。
- **Workspace**: 由一个所有者拥有、属于一个项目、具有当前状态和数据保留策略的持久工作环境。
- **Workspace Volume**: 与运行环境分离的用户持久数据资源，具有所有者、项目、容量、工作区绑定、
  保留状态和额度影响；可在同项目内由同一所有者重新绑定，所有者离开项目后由项目授权人员处理。
- **Workspace Operation**: 对一个工作区发起的可追踪生命周期请求，具有唯一操作标识、当前状态、
  终态结果、安全失败信息、请求去重关系及显式重试关联关系。
- **Audit Record**: 对安全或业务重要行为的不可篡改追踪证据。

### Security, Reliability & Operations *(mandatory when affected)*

- **Authorization**: 所有工作区与项目成员操作必须同时验证操作职责和目标项目/资源范围；未经授权
  的访问不得泄露资源存在性。项目普通成员只可管理自己的工作区；项目查看者仅可在授权范围内只读；
  项目负责人和项目操作人员仅可在明确授权的项目范围内处理成员工作区。任何用户不得访问其他项目的
  成员、工作区、操作历史或审计信息。
- **Durability & recovery**: 生命周期操作必须保持可追踪，重复请求与重复后台处理不得产生重复结果；
  发生可恢复失败时，授权用户可在不丢失用户数据的前提下重试。已产生部分外部副作用的操作必须通过
  协调流程到达明确状态，不得盲目重复创建资源。
- **Compatibility**: 已发布的工作区状态、操作追踪信息、权限含义和数据保留选择必须在后续功能扩展中
  保持兼容，除非提供明确迁移与恢复安排。
- **Operational evidence**: 每个用户请求、后续处理、操作记录和审计记录必须可通过稳定关联信息
  串联；创建、删除、重置、项目成员变更、跨用户操作、管理操作及访问拒绝均必须有审计证据；
  运行状况必须能够被运营人员辨识。
- **Trace and health behavior**: 用户请求与后续处理必须可关联；服务必须提供可供运营人员检查的存活
  状态与就绪状态信息。
- **Applicability gate**: MVP Gate；本功能必须满足 CPU 工作区闭环、基础访问控制、持久数据策略、
  操作追踪和核心验证证据要求。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在受控单节点测试环境、工作区配置所需运行环境可用且资源充足的条件下，连续进行
  不少于 20 次 CPU 工作区创建，至少 19 次在 3 分钟内进入可用或明确失败终态。
- **SC-002**: 100% 的停止后重启与默认重置验收场景保留用户数据；删除时的数据最终状态与用户选择
  一致。
- **SC-003**: 在双项目隔离验收中，100% 的跨项目或无所有权访问请求被拒绝，且响应不暴露目标资源
  是否存在。
- **SC-004**: 在重复提交、后台消息重复处理和并发生命周期操作验收中，100% 的相同请求只产生一个
  可追踪操作，不产生重复资源或重复业务副作用，且旧结果不会覆盖新状态。
- **SC-005**: 100% 的创建、删除、重置、成员变更、跨用户操作和访问拒绝验收场景具有可查询的审计证据。
- **SC-006**: 对预先定义的可恢复故障注入场景，系统能够通过自动重试、显式重试或协调流程到达明确
  终态，且不产生重复资源或持久数据丢失。

## Assumptions

- 首批交付运行在一个受控的开发环境中，只提供 CPU 工作区。
- 用户通过既有统一身份入口登录；平台可取得稳定的外部身份标识。
- 工作区配置由平台或项目负责人预先批准，用户不直接定义资源量或运行环境。
- 项目负责人负责项目成员和额度的日常管理；平台管理员负责全局配置与异常处置。
- 工作区数据保留默认面向个人开发数据，不替代正式备份或长期归档服务。
- 本规格的成功标准以功能验收环境为准；更大规模、多租户和生产运营要求在相应 Applicability Gate
  达成前不被视为本功能的交付范围。
