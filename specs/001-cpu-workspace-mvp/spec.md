# Feature Specification: 昇腾算子远程实训平台一期 MVP

**Feature Branch**: `001-cpu-workspace-mvp`

**Created**: 2026-07-22

**Status**: Approved

**Input**: 以《香橙派昇腾算子远程实训平台技术方案 V2》为一期交付基线：
为高校昇腾 C 算子实训提供多班级隔离、持久化加速工作区、三端远程接入、终端操作采集、
基础门户、资源配额与审计能力。

## Scope

本功能交付一期可用的昇腾算子远程实训平台：受控昇腾算力节点上的加速工作区全生命周期、
多班级成员与资源隔离、培训周期持久数据、浏览器/WebIDE/SSH/本地开发接入、终端操作采集、
教师可见的班级级进度与日志、节点与工作区基础运维可见性。平台管理员管理全局规则、班级、
资源配置和节点；教师在所授班级内查看学生环境与操作记录；学生只访问本人当前班级资源。

一期同时提供 CPU 与昇腾 NPU 两类工作区配置。两类工作区共享生命周期、操作追踪、持久数据和
班级隔离语义，但班级额度、调度资格、资源使用统计和验收必须分别核算。

培训周期结束时，系统默认停止学生工作区、保留其数据并限制学生继续操作；授课教师或授权管理员
可按班级批量导出资料，随后仅由授权管理员发起可审计的显式清理。平台还支持由授权 API 接受的
长期任务；每个长期任务必须使用独立任务工作区，不得复用或读取学生个人工作区，并可按班级范围
向被授权主体分配访问或管理权限。

一期明确不包括：课程与标准化实验模板、教师远程接管或终端录像复盘、作业提交与自动评分、
镜像构建流水线和安全扫描、细粒度 vNPU 配额、多地域多集群调度、教务对接、
快照/检查点恢复与移动端看板，以及长期任务对校内或公网调用方提供服务 API。后续功能不得在没有迁移方案的情况下改变本功能已经发布的
工作区状态含义、操作追踪行为、权限语义、数据保留选择或审计记录语义。

## Clarifications

### Session 2026-07-22

- Q: 默认重置完成后，工作区应处于什么状态？ → A: 重新准备运行环境，清除非持久化运行内容并保留持久数据；成功后工作区进入“运行中”。
- Q: 创建操作进行期间，用户失去班级成员资格、创建权限或可用额度时，应如何处理？ → A: 无论操作进度如何都立即终止，并尝试删除本次操作产生的所有资源和数据。
- Q: 删除工作区并选择“保留数据”后，保留的数据应如何处理？ → A: 数据继续归属原班级和原所有者并计入班级额度；所有者可将其绑定到同班级内自己的新工作区或单独删除，所有者离开班级后由班级授权人员处理。
- Q: 创建请求被接受后，所选工作区配置被停用或撤销时，应如何处理？ → A: 普通停用只阻止新请求，已接受的创建继续；安全或合规撤销立即终止创建并清理本次操作产生的资源和数据。
- Q: 用户选择“同时删除数据”，但运行环境已删除而持久数据删除失败时，应如何表示和恢复？ → A: 删除操作持续保持执行中并自动重试，直到数据删除成功，不允许用户发起显式重试。
- Q: 同一所有者在同一班级中提交重复工作区名称时，应如何处理？ → A: 未删除工作区及进行中的创建请求名称必须唯一；重复提交时拒绝并提示更换名称，已删除工作区不占用名称。

### Session 2026-07-23

- Q: 老师提供的《香橙派昇腾算子远程实训平台技术方案 V2》应如何影响当前功能范围？ → A: 直接将 `001` 扩大为完整昇腾实训一期，并重新生成其计划、数据模型、契约和前端原型。
- Q: “班级”与既有“项目”应如何建模？ → A: `Class` 替代 `Project`，成为成员、额度、工作区、存储与日志的最小隔离范围；后续如需项目概念，采用迁移方式新增。
- Q: 一期工作区应提供哪些算力类型？ → A: 同时提供 CPU 与昇腾 NPU 两类工作区 Profile，由班级额度分别约束。
- Q: 一期应采用何种角色模型？ → A: 在平台管理员、授课教师、实训学生之外，加入助教/班级操作人员和班级查看者等委派角色，并提供 RBAC 角色与角色绑定的增删改查接口，便于后续扩展。
- Q: 一期终端操作采集采用何种范围？ → A: 采集 WebIDE 与 SSH（包括本地 VS Code Remote-SSH 会话）的输入、标准输出和错误输出；教师可按本班级/学生检索，实时查看与培训周期归档均可用，并过滤密码、密钥等敏感信息。
- Q: 培训周期结束后的工作区与持久数据应如何处置？ → A: 默认停止学生工作区、保留数据并限制学生继续操作；支持按班级批量导出，授权管理员确认后显式清理。长期 API 任务使用可分配权限的独立任务工作区，与学生个人工作区隔离。
- Q: 长期任务的“API 暴露”应采用何种边界？ → A: 一期只提供受控控制面 API 提交、查询和管理长期任务；任务本身不提供可调用 API。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建并管理自己的算力工作区 (Priority: P1)

作为班级成员，我希望创建一个经过批准的 CPU 或昇腾 NPU 工作区，并能查看其进度、停止、重新启动、
重置和删除它，以便在不重新配置环境的情况下持续完成开发或实验任务。

**Why this priority**: 这是平台的最小用户价值闭环；没有可控且可恢复的工作区，后续教学、
算力和终端能力没有可靠的承载对象。

**Independent Test**: 为一个已授权班级成员创建工作区，完成启动、停止、重启、重置和删除，
并验证每一步有可见结果、失败可理解、数据保留行为符合用户选择。

**Acceptance Scenarios**:

1. **Given** 班级成员有相应算力类型的可用额度，**When** 选择一个批准的 CPU 或昇腾 NPU 工作区配置并提交创建，
   **Then** 系统提供唯一的操作追踪信息，并在成功后显示可用工作区。
2. **Given** 工作区正在运行，**When** 用户停止它，**Then** 系统显示已停止且用户数据仍可在随后
   启动时使用。
3. **Given** 工作区已停止或运行中，**When** 用户执行默认重置，**Then** 系统重新准备运行环境、清除
   非持久化运行内容、保留独立的用户持久数据，并在成功后将工作区置为运行中。
4. **Given** 用户提交相同的创建或生命周期请求两次，**When** 系统处理请求，**Then** 只产生一个
   对应的工作区或生命周期操作。
5. **Given** 创建操作尚未完成，**When** 用户失去班级成员资格、创建权限或可用额度，**Then** 系统
   立即终止该操作，尝试删除本次操作产生的全部资源和数据，并向授权人员显示最终处理结果。
6. **Given** 用户删除工作区并选择保留数据，**When** 工作区删除完成，**Then** 数据保持原班级和
   原所有者归属并继续计入班级额度，所有者可将其绑定到同班级内自己的新工作区或单独删除。
7. **Given** 创建请求已经被接受，**When** 所选配置被普通停用，**Then** 该创建可继续且系统拒绝
   后续使用该配置的新请求。
8. **Given** 创建请求尚未完成，**When** 所选配置因安全或合规原因被撤销，**Then** 系统立即终止
   创建并清理本次操作产生的资源和数据。
9. **Given** 用户选择同时删除工作区和持久数据，且运行环境已删除但数据删除失败，**When** 系统
   继续处理删除，**Then** 该删除操作保持执行中并自动重试至数据删除成功，用户不得另行发起该
   操作的显式重试。
10. **Given** 同一所有者在同一班级中已有同名的未删除工作区或同名创建请求正在进行，**When** 用户
    提交该名称的创建请求，**Then** 系统拒绝请求并提示用户更换名称；已删除工作区不占用名称。

### User Story 2 - 按班级范围管理访问与角色 (Priority: P2)

作为授课教师或被授权的助教，我希望管理班级成员职责并查看或处理班级内工作区，以便在不暴露
其他班级数据的情况下支持团队成员。

**Why this priority**: 基础访问边界是共享教学和班级环境安全运行的前提；仅有登录无法防止
跨班级或跨用户访问。

**Independent Test**: 创建两个班级和不同成员，在其中一个班级执行成员与工作区管理操作，验证
另一个班级成员无法查看、修改或推测这些资源。

**Acceptance Scenarios**:

1. **Given** 授权管理员拥有角色管理权限，**When** 增加、修改、停用或删除角色、权限定义或角色绑定，
   **Then** 变更仅在授权范围内生效、保留兼容的历史审计信息并被记录。
2. **Given** 授课教师、助教或经授权的班级操作人员，**When** 查看或处理班级内成员的工作区，**Then** 系统
   显示操作主体、目标与结果。
3. **Given** 实训学生、班级查看者或另一班级成员，**When** 尝试执行超出其职责或不属于自己的工作区操作，**Then** 系统拒绝请求且
   不泄露资源是否存在。

---

### User Story 3 - 跟踪失败并安全恢复 (Priority: P3)

作为用户或班级负责人，我希望了解工作区操作是否失败、为何失败以及何时可以重试，以便从暂时性
故障中恢复而不丢失数据或产生重复资源。

**Why this priority**: 工作区依赖外部运行环境；可见的失败、重试和恢复能力将人工排障从常规流程中
移除。

**Independent Test**: 在创建、启动、停止或删除过程中模拟暂时性失败，验证系统保留操作历史、
显示可操作的状态，并在安全重试后不产生重复资源或错误覆盖新状态。

**Acceptance Scenarios**:

1. **Given** 一个工作区操作无法完成，**When** 系统确认失败，**Then** 用户可看到失败状态、可理解的
   原因和下一步操作。
2. **Given** 一个可恢复的失败操作，**When** 授权用户重试，**Then** 系统继续或重新执行该操作而不创建
   重复资源。
3. **Given** 用户已发起新操作，**When** 较早操作晚到的结果返回，**Then** 系统保留较新的工作区状态。
4. **Given** 调用者重复提交同一业务请求，**When** 系统收到重复提交，**Then** 系统返回原操作的追踪
   信息和结果，不产生新的副作用。
5. **Given** 一个允许恢复的失败操作，**When** 授权用户发起显式重试，**Then** 系统创建关联原失败
   操作的新操作记录，而非重复使用原请求。

---

### User Story 4 - 查看班级终端操作与实训进度 (Priority: P1)

作为授课教师或被授权助教，我希望按本班级和学生查看实训终端操作及输出，以便追踪实训进度、
定位常见问题并提供教学支持。

**Why this priority**: 老师方案将终端操作采集列为一期教学分析的硬性能力；没有可检索的过程证据，
平台只能提供算力，无法支持实训过程管理。

**Independent Test**: 学生分别通过 WebIDE 和 SSH（含本地 VS Code Remote-SSH）完成终端操作，
验证授课教师只能查询本人班级的已过滤输入、标准输出和错误输出，并能查看实时记录与培训周期历史。

**Acceptance Scenarios**:

1. **Given** 实训学生在本人班级工作区通过 WebIDE 或 SSH 执行终端操作，**When** 系统接收输入、
   标准输出或错误输出，**Then** 系统以班级、学生、工作区和时间关联该记录。
2. **Given** 授课教师或被授权助教，**When** 按本人班级和学生查询终端操作，**Then** 系统仅展示其
   授权范围内已过滤的实时记录和培训周期历史。
3. **Given** 终端输入或输出包含密码、密钥或其他预定义敏感内容，**When** 系统采集该内容，**Then**
   对用户可见与长期保存的记录均不包含该敏感内容。

---

### User Story 5 - 以独立工作区执行长期 API 任务 (Priority: P2)

作为被授权的任务调用者，我希望通过 API 提交长期任务并为其分配独立工作区和班级范围权限，
以便任务长期执行而不影响或暴露学生个人实训环境。

**Why this priority**: 长期任务与交互式学生实训的资源、数据和授权边界不同；复用学生工作区会造成
越权访问、资源干扰和数据泄露风险。

**Independent Test**: 为一个班级提交长期任务，验证系统创建独立任务工作区；仅被授予该任务范围
权限的主体可查询或管理它，且任务无法读取学生工作区、数据或终端记录。

**Acceptance Scenarios**:

1. **Given** 调用主体拥有班级范围内的长期任务提交权限，**When** 通过受控 API 提交任务，**Then**
   系统为该任务创建或分配独立工作区、稳定操作追踪信息和班级归属。
2. **Given** 一个长期任务工作区，**When** 未被授予其班级和任务范围权限的主体尝试访问，**Then**
   系统拒绝请求且不泄露任务或工作区是否存在。
3. **Given** 长期任务与学生工作区同属一个班级，**When** 任务运行或被查询，**Then** 任务的资源、
   数据和终端记录仍与学生个人工作区隔离。

### Edge Cases

- 用户在已有未完成操作时再次请求冲突的生命周期操作。
- 同一所有者在同一班级中提交与未删除工作区或进行中的创建请求同名的工作区；系统必须拒绝请求
  并提示更换名称，已删除工作区不占用名称。
- 用户在创建操作期间失去班级成员资格、创建权限或可用额度；操作必须立即终止，并清理本次操作
  已产生的全部资源和数据。
- 用户选择删除工作区但要求保留其数据；数据继续归属原班级和原所有者并计入班级额度，可由所有者
  重新绑定到同班级内自己的新工作区或单独删除，所有者离开班级后由班级授权人员处理。
- 用户选择删除工作区并同时删除数据；系统必须分别显示运行环境与持久数据的最终处理结果。
- 运行环境已删除但持久数据删除失败；删除操作保持执行中并自动重试至数据删除成功，不允许用户
  对该删除操作发起显式重试。
- 系统在外部环境已完成部分处理后中断或重启。
- 外部请求超时但资源可能已创建成功，或后台消息被重复处理；系统必须协调现有结果而不重复创建
  资源或产生重复业务副作用。
- 较旧工作区代次的结果在新操作之后返回；较旧结果不得覆盖较新状态。
- 班级负责人尝试访问另一班级的工作区，或普通成员猜测其他用户的资源标识；请求必须被拒绝且
  不泄露资源存在性。
- 用户对不可恢复失败重复重试；系统必须拒绝该重试并保留原失败原因。
- 用户尝试用无效、过期或不再被批准的工作区配置创建环境；普通停用只阻止新请求，安全或合规
  撤销还会立即终止尚未完成的已接受创建并清理本次操作产生的资源和数据。
- 终端记录包含敏感内容、来自另一班级的查询或输入输出流中断；系统必须先过滤敏感内容、拒绝
  范围外查询，并保留可关联的采集缺口或恢复证据。
- 长期任务被重复提交、分配权限被撤销或调用主体跨班级访问；系统必须以任务工作区的独立身份、
  班级归属和授权范围处理，不得复用学生工作区或泄露其存在性。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将每个外部身份稳定映射到唯一的平台用户；电子邮件地址 MUST NOT 作为
  唯一或权威的用户身份依据。
- **FR-002**: 系统 MUST 支持平台管理员、授课教师、助教/班级操作人员、实训学生和班级查看者等
  职责；工作区所有权 MUST 是独立于班级职责的资源关系。所有职责 MUST 仅在其被授予的班级范围内
  生效。
- **FR-020**: 系统 MUST 提供受授权管理员使用的 RBAC 角色、权限和角色绑定的查询、创建、修改、
  停用与删除能力；角色或绑定变更 MUST 不突破班级范围、不得静默改变既有资源授权，并形成可查询
  审计证据。
- **FR-021**: 系统 MUST 采集 WebIDE 与 SSH（包括本地 VS Code Remote-SSH 会话）的终端输入、
  标准输出和错误输出，并以班级、用户、工作区和时间进行关联；授课教师和被授权助教仅可查看
  其班级范围内的实时记录与培训周期历史。
- **FR-022**: 系统 MUST 在终端记录可见、检索或归档前过滤密码、密钥和其他预定义敏感内容；采集
  中断、过滤或恢复结果 MUST 可关联并形成运营证据，终端记录访问 MUST 形成审计证据。
- **FR-023**: 培训周期结束时，系统 MUST 默认停止学生工作区、保留数据并限制学生继续操作；
  授课教师或授权管理员 MUST 能按班级批量导出资料，且仅授权管理员可发起可审计的显式清理。
- **FR-024**: 系统 MUST 支持由授权 API 提交的长期任务，为每个任务创建或分配独立任务工作区、
  班级归属、稳定操作追踪信息和可撤销的任务范围授权；任务工作区 MUST NOT 复用、读取或写入
  学生个人工作区、持久数据或终端记录。一期任务 API MUST 仅用于受控控制面提交、查询和管理，
  任务本身 MUST NOT 对校内或公网调用方提供服务 API。
- **FR-003**: 系统 MUST 仅允许拥有创建权限且相应资源额度可用的班级成员，从平台批准且当前班级
  已启用的 CPU 或昇腾 NPU 工作区配置中选择并创建自己的工作区；用户 MUST NOT 任意指定运行环境、
  资源量或特权能力。
- **FR-004**: 系统 MUST 为创建、启动、停止、重置和删除请求生成稳定、唯一且可查询的操作标识，
  并显示等待、执行、成功或失败等当前状态、终态结果及安全的失败原因。
- **FR-005**: 系统 MUST 仅允许合法的工作区状态转换；冲突的生命周期操作 MUST NOT 同时改变或破坏
  工作区状态，且系统 MUST 告知用户现有操作尚未完成。
- **FR-006**: 系统 MUST 在停止后保留用户持久数据；默认重置 MUST 重新准备运行环境、清除非持久化
  运行内容、保留独立的用户持久数据，并在成功后使工作区进入运行中；删除请求 MUST 明确展示数据
  保留或删除的选择与结果。
- **FR-007**: 系统 MUST 要求用户同时具备相应操作权限和目标资源的所有权或班级范围授权；界面隐藏
  不得被视为访问控制。
- **FR-008**: 系统 MUST 将同一调用者对同一业务请求的重复提交视为同一操作，返回同一操作结果，
  且不得产生重复资源或重复副作用；用户对允许恢复的失败操作发起显式重试时，系统 MAY 创建新的
  操作记录，但 MUST 与原失败操作关联。
- **FR-009**: 系统 MUST 记录工作区创建、删除、重置、跨用户操作、班级成员变更和拒绝访问的主体、
  目标、时间、结果和关联追踪信息。
- **FR-010**: 系统 MUST 在工作区操作失败时保留失败历史、提供不暴露凭证、堆栈、数据库信息或
  运行环境敏感细节的原因，并仅在安全时允许授权用户重试或恢复；不可恢复失败的重复重试 MUST
  被拒绝并保留原失败原因。
- **FR-011**: 系统 MUST 只允许使用平台批准的工作区配置；用户不得指定任意运行环境或获得超出其
  配额的资源。
- **FR-012**: 系统 MUST 允许班级负责人查看班级内工作区的状态和操作历史，并禁止其访问其他班级
  的同类信息。
- **FR-013**: 创建操作进行期间，如果调用者失去班级成员资格、创建权限或可用额度，系统 MUST
  立即终止该操作，并尝试删除本次操作产生的全部资源和数据；清理进度、失败和最终结果 MUST
  保持可查询并形成审计证据。
- **FR-014**: 删除工作区并选择保留数据后，系统 MUST 保持数据的原班级和原所有者归属并继续计入
  班级额度；所有者 MUST 能够将其绑定到同班级内自己的新工作区或单独删除；所有者离开班级后，
  该数据 MUST 由班级授权人员处理。
- **FR-015**: 工作区配置被普通停用时，系统 MUST 拒绝使用该配置的新创建请求，并 MUST 允许已经
  接受的创建继续；配置因安全或合规原因被撤销时，系统 MUST 立即终止尚未完成的相关创建，清理
  本次操作产生的资源和数据，并使终止、清理及最终结果保持可查询和可审计。
- **FR-016**: 当用户选择同时删除数据且运行环境已删除、持久数据删除失败时，系统 MUST 保持删除
  操作处于执行中并自动重试至数据删除成功；用户 MUST NOT 对该删除操作发起显式重试，且自动处理
  过程必须可查询、可关联并可审计。在数据删除成功前，持久数据 MUST 保持原所有权、班级归属和
  额度影响。
- **FR-017**: 系统 MUST 为已接受但尚未执行、已执行但尚未产生外部副作用、以及已产生部分外部
  副作用的操作分别提供可追踪的终止或协调规则；平台进程中断、外部请求超时或消息重复处理后，
  系统 MUST 协调实际结果至明确状态，不得盲目重复创建资源。
- **FR-018**: 较早操作或较旧工作区代次的晚到结果 MUST NOT 覆盖较新的工作区状态。
- **FR-019**: 同一所有者在同一班级内，未删除工作区及进行中的创建请求的名称 MUST 唯一；系统
  MUST 拒绝重复名称的创建请求并提示用户更换名称。已删除工作区 MUST NOT 占用名称。

### Key Entities *(include if feature involves data)*

- **User**: 具有稳定平台身份、状态与平台级职责的个人或受控自动化主体。
- **Class**: 成员、额度、工作区配置和资源归属的最小管理范围。
- **Class Membership**: 用户在班级内的职责和有效访问范围。
- **Role / Permission / Role Binding**: 可演进的 RBAC 定义与绑定关系；角色授予一组权限，绑定将角色
  限定到平台或班级范围，所有变更保留审计证据。
- **Workspace Profile**: 平台批准的 CPU 或昇腾 NPU 工作区选项，包含算力类型、用户可见名称、
  资源额度和可用状态。
- **Workspace**: 由一个所有者拥有、属于一个班级、具有当前状态和数据保留策略的持久工作环境。
- **Workspace Volume**: 与运行环境分离的用户持久数据资源，具有所有者、班级、容量、工作区绑定、
  保留状态和额度影响；可在同班级内由同一所有者重新绑定，所有者离开班级后由班级授权人员处理。
- **Workspace Operation**: 对一个工作区发起的可追踪生命周期请求，具有唯一操作标识、当前状态、
  终态结果、安全失败信息、请求去重关系及显式重试关联关系。
- **Audit Record**: 对安全或业务重要行为的不可篡改追踪证据。
- **Terminal Operation Record**: 与班级、学生、工作区和时间关联的已过滤终端输入、标准输出或
  错误输出记录，支持实时查询与培训周期归档。
- **Long-Running Task / Task Workspace**: 由授权 API 接受的长期任务及其独立工作区，具有班级归属、
  调用主体、可撤销任务范围授权、资源额度和操作追踪；与学生个人工作区隔离。

### Security, Reliability & Operations *(mandatory when affected)*

- **Authorization**: 所有工作区、班级成员与 RBAC 管理操作必须同时验证操作职责和目标班级/资源范围；
  未经授权的访问不得泄露资源存在性。实训学生只可管理自己的工作区；班级查看者仅可在授权范围内
  只读；授课教师、助教和班级操作人员仅可在明确授权的班级范围内处理成员工作区。任何用户不得访问
  其他班级的成员、工作区、操作历史或审计信息。
- **Long-running task isolation**: 长期任务必须以独立任务工作区和独立授权范围运行；任务 API 的
  调用主体、班级范围、任务范围、权限授予/撤销和访问拒绝均必须可追踪并审计；一期不允许任务
  自行对校内或公网调用方暴露服务接口。
- **Durability & recovery**: 生命周期操作必须保持可追踪，重复请求与重复后台处理不得产生重复结果；
  发生可恢复失败时，授权用户可在不丢失用户数据的前提下重试。已产生部分外部副作用的操作必须通过
  协调流程到达明确状态，不得盲目重复创建资源。
- **Compatibility**: 已发布的工作区状态、操作追踪信息、权限含义和数据保留选择必须在后续功能扩展中
  保持兼容，除非提供明确迁移与恢复安排。
- **Operational evidence**: 每个用户请求、后续处理、操作记录和审计记录必须可通过稳定关联信息
  串联；创建、删除、重置、班级成员与角色变更、跨用户操作、管理操作及访问拒绝均必须有审计证据；
  运行状况必须能够被运营人员辨识；终端采集范围、过滤结果和采集缺口必须可供授权运营人员检查。
- **Trace and health behavior**: 用户请求与后续处理必须可关联；服务必须提供可供运营人员检查的存活
  状态与就绪状态信息。
- **Applicability gate**: MVP Gate；本功能必须满足 CPU 与昇腾 NPU 工作区闭环、基础访问控制、
  持久数据策略、操作追踪和核心验证证据要求。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在受控单节点测试环境、工作区配置所需运行环境可用且资源充足的条件下，连续进行
  不少于 20 次 CPU 工作区创建，至少 19 次在 3 分钟内进入可用或明确失败终态。
- **SC-002**: 100% 的停止后重启与默认重置验收场景保留用户数据；删除时的数据最终状态与用户选择
  一致。
- **SC-003**: 在双班级隔离验收中，100% 的跨班级或无所有权访问请求被拒绝，且响应不暴露目标资源
  是否存在。
- **SC-004**: 在重复提交、后台消息重复处理和并发生命周期操作验收中，100% 的相同请求只产生一个
  可追踪操作，不产生重复资源或重复业务副作用，且旧结果不会覆盖新状态。
- **SC-005**: 100% 的创建、删除、重置、成员变更、跨用户操作和访问拒绝验收场景具有可查询的审计证据。
- **SC-006**: 对预先定义的可恢复故障注入场景，系统能够通过自动重试、显式重试或协调流程到达明确
  终态，且不产生重复资源或持久数据丢失。

## Assumptions

- 首批交付运行在受控单节点开发环境，同时提供 CPU 与昇腾 NPU 工作区；两类资源使用独立的班级配额、调度资格与验收统计。
- 用户通过既有统一身份入口登录；平台可取得稳定的外部身份标识。
- 工作区配置由平台或班级负责人预先批准，用户不直接定义资源量或运行环境。
- 班级负责人负责班级成员和额度的日常管理；平台管理员负责全局配置与异常处置。
- 工作区数据保留默认面向个人开发数据，不替代正式备份或长期归档服务。
- 本规格的成功标准以功能验收环境为准；更大规模、多租户和生产运营要求在相应 Applicability Gate
  达成前不被视为本功能的交付范围。
